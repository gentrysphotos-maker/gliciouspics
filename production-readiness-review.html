3. Launch Blockers Critical

These must be resolved before accepting real money or orders.

Mock checkout must be replaced

pages/cart.html shows a modal saying "Thank you for your mock order" and clears the cart. There is no Stripe, PayPal, or any payment flow.

Product not-found shows wrong product

All three product templates (product-standard.html, product-panorama.html, product-aerial.html) fall back to the first product in their category when an ?id= is missing or invalid — instead of showing the "Coming Soon" state. A bad link could let a customer order the wrong print.

Fix: Remove the fallback lines (~788 in each product template). Only call renderProduct() when a matching ID exists; otherwise call renderNotFound().

No server-side price validation

Cart items store price in localStorage. A user can open DevTools and set any price. Production checkout must re-validate line items and totals on the server against products.json (or a database) before creating a payment session.

Contact form is non-functional

js/main.js line 74 notes it is a placeholder. Submissions show a fake success message but never send email. Customers cannot reach you through the form.

4. Payments & Print Fulfillment

Payment integration — recommended approach

For a static site selling physical prints, Stripe Checkout is the best fit:

  1. Add a serverless function (Vercel/Netlify/Cloudflare) that accepts cart items
  2. Server looks up real prices from product data
  3. Server creates a Stripe Checkout Session with line items + shipping
  4. Redirect customer to Stripe-hosted checkout (PCI-compliant, handles cards/Apple Pay)
  5. Stripe webhook confirms payment → trigger fulfillment

Do not embed Stripe secret keys in frontend JS. Only publishable keys belong in the browser.

Print fulfillment — choose a workflow

Your product copy references ChromaLuxe metal and lustre paper — typical of pro labs like Bay Photo, WHCC, or similar. Options:

ApproachEffortBest for
Manual fulfillment
Stripe webhook → email you order details → you submit to lab manually		 Low Soft launch, low volume
Printful / Gelato API
Automated order routing to print partner		 Medium Hands-off scaling; verify they offer your exact media (metal, lustre)
WHCC / Bay Photo lab portal
Upload high-res files, manual or API order entry		 Medium–High Maximum print quality control (likely your current lab)
Shopify / Snipcart overlay Medium If you want hosted commerce without building backend

Shipping

  • Cart hardcodes "FREE" shipping — FAQ says free US + many countries but custom quotes elsewhere
  • No shipping address collection exists today
  • Stripe Checkout can collect shipping address; you'll need shipping rate rules or flat rates per region
  • International customs/duties not mentioned — add to FAQ if selling abroad

Order communications

  • Order confirmation email (to customer)
  • Order notification email (to you)
  • Shipping notification with tracking (from lab or manual)
  • FAQ promises confirmations within 24–48 hours — infrastructure must support this

Material naming inconsistency

products.json uses "Chromaluxe Metal" and "Lustre Paper". FAQ describes "High Gloss Chromaluxe Aluminum" and "Mid Gloss Chromaluxe Aluminum" (custom order). Align customer-facing names across shop dropdowns, cart receipts, and FAQ to avoid confusion.

5. Images & Performance

Hero slideshow — heavy images

Homepage hero slides use w_2560,q_100 (5 full-quality backgrounds, ~several MB total). Gallery thumbnails correctly use f_auto,q_auto. Hero should match: f_auto,q_auto,w_1920 (or responsive srcset via Cloudinary).

About photo uses q_100

index.html profile image: w_1600,q_100 — switch to q_auto.

Product page loads entire catalog

Each product page fetches all of products.json (200 KB) to render one product + 4 suggestions. Options:

  • Split JSON by category (products/landscapes.json, etc.)
  • Generate per-product static pages at build time
  • Cache JSON in sessionStorage after first fetch
  • Add Cache-Control headers on deploy (JSON rarely changes)

Missing Cloudinary preconnect

Google Fonts have preconnect but Cloudinary does not. Add:

<link rel="preconnect" href="https://res.cloudinary.com" crossorigin>

Font loading

Two Google Font families with multiple weights load on every page. Consider: self-hosting WOFF2 subsets, reducing weight variants, or font-display: swap in the Google URL.

Good practices already in place

  • loading="lazy" on gallery and below-fold images
  • Cloudinary f_auto serves WebP/AVIF where supported
  • Passive scroll listener in main.js
  • Thumbnail preloading on product image switch

6. Code Quality & Organization

Major duplication

  • Nav + footer copied across ~15 HTML files (~50 lines each). Any nav change requires editing every file.
  • Three product templates (product-standard, product-panorama, product-aerial) share ~95% identical HTML, CSS, and JS (~800 lines each).
  • Product page inline JS (~300 lines) duplicated 3× — should be js/product.js.
  • FAQ accordion JS duplicates product accordion pattern.

Recommendation: Introduce a minimal build step (11ty, Vite, or simple Node script) for includes/templates, OR migrate to a lightweight framework. Even a 20-line Node include script would cut maintenance dramatically.

Gallery pages should be generated

130 products × manual HTML links is fragile. You already have scratch/check_missing_*.js scripts — promote these to a npm run build:galleries script that reads products.json and regenerates gallery markup. This eliminates sync drift (2 products currently missing from galleries).

Redundant link paths

Gallery pages inside pages/ link to ../pages/product-standard.html instead of product-standard.html. Works today but is confusing and error-prone.

Mixed JS style

main.js uses const/let and arrow functions; cart.js and inline scripts use var and function. Pick one style.

Size/material dropdowns not linked

On product pages, size and material dropdowns are independent. Selecting a size that doesn't exist for a material still shows in the dropdown, leading to "combination not available" alerts. Filter available sizes when material changes (and vice versa).

Lightbox limitation

initLightbox() skips items that are <a> tags. All gallery items are links, so the lightbox never activates on gallery pages. Either remove dead lightbox code or add a "quick preview" mode separate from the product link.

Case-sensitive product IDs

Japan-itsukushima-shrine-travel-photography uses a capital J in the ID. Works on case-sensitive servers but is inconsistent with all-lowercase kebab-case IDs elsewhere.

products.json size

At 200 KB the file is manageable but will grow. Consider stripping internal-only fields from the public JSON and keeping high-res source URLs server-side only.

7. Design & UX

Strengths

  • Cohesive dark editorial aesthetic suits fine art photography
  • Clear category navigation and "View + Buy Print" CTAs on galleries
  • Product pages have mockup thumbnails, material accordions, and "You Might Also Like"
  • Sticky product info column on desktop
  • Cart page layout is clean with qty controls and summary panel

Issues to address

  • alert() for validation — replace with inline error messages near the Add to Cart button
  • Instagram handle mismatch — links use instagram.com/g.liciouspics but contact shows @g.liciouspics
  • Facebook link is a placeholder — facebook.com with no page path; remove or add real URL
  • No favicon — browser tab shows generic icon
  • No 404 page — broken URLs get host default
  • Cart lacks "added to cart" toast on product pages (button text changes briefly — easy to miss)
  • No breadcrumb navigation on product pages (Home → Landscapes → Print Title)
  • Slideshow has no pause on hover or reduced-motion support
  • External links missing rel="noopener noreferrer" on target="_blank" links

Accessibility gaps

  • No skip-to-content link
  • No :focus-visible styles for keyboard navigation
  • Dropdown trigger is a <span>, not a button — not keyboard/screen-reader friendly
  • Accordions update aria-expanded (good) but FAQ/product accordions close all others on open — consider allowing multiple open
  • Hero slideshow images are CSS backgrounds — no alt text for screen readers
  • Color contrast on --muted (#6a6a6a) text may fail WCAG AA on dark backgrounds

9. Legal & Trust

Required before selling online:

  • Privacy Policy — required if collecting emails, addresses, payment info (GDPR/CCPA)
  • Terms of Service — sale terms, custom print policies
  • Refund / Return Policy — FAQ mentions no returns + 6-hour cancel window; needs a dedicated page linked in footer/checkout
  • Cookie notice — if adding analytics (Google Analytics, Meta Pixel)
  • Business entity info — consider adding business name/location in footer for trust
  • Sales tax — Stripe Tax can handle US sales tax collection if needed

10. Security

  • Cart price tampering — client-side prices are not trustworthy (see Blockers)
  • No Content-Security-Policy — add via hosting headers to restrict script sources
  • No security headers — configure X-Frame-Options, X-Content-Type-Options, Referrer-Policy on deploy
  • Cloudinary cloud name is public — normal for delivery URLs; ensure upload API keys are never in frontend code
  • Stripe webhooks — must verify webhook signatures server-side
  • Rate limiting — on contact form and checkout session creation endpoints
